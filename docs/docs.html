4. Tehnologii si concepte folosite

Urmatoarele tehnologii si concepte au fost folosite:

  • NodeJS - Pentru a mentine acelasi limbaj de programare atat pe frontend cat si pe backend am folosit runtime-ul NodeJS
  • Cloud Firestore - Am ales o baza de data nerelationala (NoSQL) in cloud pentru a nu exista overhead-ul de a folosi un Object Relational Mapper (ORM), in plus, faptul ca am folosit o baza de data in cloud a usurat procesul de dezvoltare deoarece am avut datele sincronizate indiferent de masina pe care rula aplicatia.
  • AJAX - Pentru a putea modifica pagina in mod dinamic s-a folosit API-ul fetch, am dorit ca datele sa fie preluate asincron de catre JavaScript pentru a nu fi nevoie de a reincarca pagina de fiecare data cand se preluau noi date.
  • CSV - Am adaugat optiunea de a putea exporta mesajele unei conversatii in format CSV.
  • JSON Web Tokens (JWT) - Aceste token-uri s-au folosit pentru a securiza API-ul astfel incat doar utilizatorii autorizati au access la anumite resurse.
  • OpenAPI Specification - Pentru a vizualiza mai usor modul de access al datelor aplicatiei am construit o specificatie al API-ului.

5. API

In contextul API-ului, datele utilizatorilor se afla sub colectia /users. Urmatoarele date apartin fiecarui utilizator:

  • avatar_link - Un link ce contine continut media de tip imagine
  • name - Numele utilizatorului
  • email - Email-ul utilizatorului, utilizat ca un identificator unic al sau
  • password - Parola utilizatorului in format hexadecimal dupa executarea algoritmului SHA256
  • admin - Camp boolean cu valoarea true care apare doar la utilizatorii de tip admin

In cadrul utilizatorilor se remarca endpoint-ul POST /users, care serveste ca un punct unde se inregistreaza utilizatorii. Pentru login se foloseste endpoint-ul /login, unde ca date sunt necesare email-ul si parola utilizatorului iar in caz de potrivire a credentialelor se va returna un JWT.

Urmatoarele colectii, subcolectii si resurse mentionate definesc modul de functionare al mesajelor. Resursa care se gaseste la endpoint-ul /chats/{adminId} reprezinta un array de ID-uri ale utilizatorilor cu care admin-ul cu ID-ul {adminId} are mesaje. Intuitiv, colectia reprezentata de endpoint-ul /chats/{adminId}/{userId} reprezinta mesajele intre admin-ul cu ID {adminId} si utilizatorul cu ID {userId}. Mai mult, resursa gasita la endpoint-ul /chats/{adminId}/{userId}/{messageId} este mesajul cu id {messageId} dintre conversatia intre admin-ul cu ID {adminId} si utilizatorul cu ID {userId}
